# OAuth 2.1 Implementation Plan

**Goal:** Each Yap instance is its own OAuth 2.1 authorization server and resource
server, per the auth spec. Two credential lanes everywhere: access keys (root
credential) and OAuth tokens (delegated, scoped). MCP clients (e.g. Claude)
connect via standard discovery with zero manual config.

**Spec reconciliation decisions:**

- **Roles**: capability grants stay ground truth. The spec's `admin` / `member` /
  `read-only` become capability masks carried by token scopes and enforced as a
  clamp over live grant resolution: `allowed = grants ∧ mask ∧ restriction`.
  Masks: `read-only` = read_items, read_files; `member` = content capabilities +
  create/edit_bundles; `admin` = no clamp (full delegation).
- **Token format**: opaque, stored hashed (same SHA-256 treatment as access
  keys). Prefixes `yap_at_` (access) / `yap_rt_` (refresh) for lane routing and
  secret scanning. Revocation is a DB row, effective immediately.
- **Default scope** when a client requests none: `role:member` — a connected
  agent can work with content but cannot mint access keys, manage roles, or
  manage spaces. The web app will explicitly request `role:admin`.
- **Credential management over the token lane** (key CRUD, connected-apps
  management) requires `role:admin`; the key lane can always. This blocks the
  privilege-escalation path token → new access key → full authority.
- **Scope threading**: AsyncLocalStorage (`src/core/authScope.ts`), established
  at the two transport entry points (REST handler wrapper, MCP tool wrapper),
  read inside capability resolution. Avoids re-typing ~40 core signatures;
  unscoped contexts (key lane, sysadmin provisioning, background sweeps)
  resolve as today.

## Pieces

1. **Schema** (both twins + generated migrations): `oauth_clients`,
   `oauth_codes` (PKCE challenge, hashed code, 60s TTL), `oauth_grants`
   (user × key × client × scope — the "connected apps" rows; `key_id` is what
   makes revoke-key-kills-grants one indexed delete), `oauth_tokens` (hashed,
   kind access/refresh, expiry, revoked_at).
2. **Config**: `YAP_OAUTH_ACCESS_TOKEN_TTL_SECONDS` (3600),
   `YAP_OAUTH_REFRESH_TOKEN_TTL_SECONDS` (30d), `YAP_OAUTH_CODE_TTL_SECONDS` (60).
3. **Core** `src/core/oauth.ts`: client registration (public clients only,
   PKCE mandatory, https-or-loopback redirect URIs with RFC 8252 loopback port
   variance), code mint/exchange, refresh rotation with reuse detection
   (reused refresh → whole grant revoked), RFC 7009 revocation, token
   authentication, scope parse/format, per-user grant list/revoke.
4. **Clamp** in `src/core/capabilities.ts`: scope check ahead of grant
   resolution and the personal-owner shortcut; new `token_scope` decision.
5. **Keys**: `rotateKey`/`deleteKey` cascade to oauth grants;
   `authenticateKeyRow` returns the key id so consent can bind grants to keys.
6. **HTTP** `src/rest/oauth.ts`: RFC 8414 + RFC 9728 well-known docs (plus
   `/mcp` path-suffix variants), RFC 7591 `POST /oauth/register`,
   `GET|POST /oauth/authorize` (consent page, authenticates by access key,
   sysadmin key refused), `POST /oauth/token` (code+PKCE, refresh),
   `POST /oauth/revoke`, `GET|DELETE /v1/oauth/grants`.
7. **Transports**: REST `handle` wrapper and MCP `authenticate` accept the
   token lane; MCP 401s carry `WWW-Authenticate: Bearer resource_metadata=…`
   (what makes MCP clients auto-discover the flow); MCP tools wrapped to
   establish the scope context.
8. **Tests**: end-to-end flow (register → consent → code → tokens → REST+MCP
   access), PKCE enforcement, read-only and space-restriction clamps, refresh
   rotation + reuse, key-revocation cascade, token-lane key-management denial,
   discovery shape, WWW-Authenticate.

## Out of scope (later phases)

Resource-restricted access keys, consent memory, DCR rate limiting and idle
client sweeping, non-loopback plain-http issuer refusal (warn only).
